## Open

- [ ] [P0] **The CMS admin surface has NO authentication** (security review, 2026-06-17). Every `/admin/*` page (`/admin/chat`, `/admin/settings/*`, `/admin/media`) and every admin API route (`/api/chat`, `/api/settings/*`, `/api/assets` GET/POST/DELETE) is reachable by anyone with no session check — a stranger can drive the AI, change content locales, and upload/delete media on a deployed CMS Worker. This is a pre-existing, CMS-wide gap (the CMS app ships no auth/session module at all, unlike PM), not introduced by any one feature; a per-route patch would be fake security since there's no session system to call. NEEDS A DECISION + an epic: how do CMS admins authenticate? Options: (a) the CMS Worker validates a PM-issued session/JWT (PM already has KV-backed sessions — share the secret/validate centrally); (b) a standalone email+password auth in the CMS per-Site D1 (mirror PM's `lib/auth`, PBKDF2 ≤100k iters per the Workers cap). Then add a `requireAdmin(request)` gate at the top of every `/api/*` admin route + a layout/redirect guard for `/admin/*` (or a Next proxy matching both — note CMS has no proxy yet, and Next-16-on-OpenNext proxy constraints from the PM i18n caveat apply). WHY user/judgment: it's an architecture choice (shared PM auth vs. standalone CMS auth) with a deploy/secret dependency. — added 2026-06-17 by D1 R2 asset gallery (security review)
