- **PM i18n is `next-intl` v4, COOKIE-BASED (no proxy/middleware), NOT path-prefix routing.** WHY: Next 16 renamed middleware→`proxy` and made it Node-runtime only; the current OpenNext Cloudflare adapter can't bundle a Node proxy, and Next 16 hard-rejects `runtime: "edge"` on a proxy ("Proxy does not support Edge runtime"). So `/[locale]` path routing builds with `next build` but FAILS `opennextjs-cloudflare build` (the deploy gate). Cookie-based passes both. Refs: workers-sdk#13755, opennextjs-cloudflare#962. Architecture: `src/i18n/routing.ts` (locale source of truth: EN/FI/ET, `localeNames`, `LOCALE_COOKIE=NEXT_LOCALE`), `src/i18n/request.ts` (`resolveLocale()` = cookie → Accept-Language → default), `createNextIntlPlugin` in `next.config.ts`, root layout wraps `NextIntlClientProvider` + `getLocale()`, `LocaleSwitcher` sets the cookie + `router.refresh()`. Catalogs: `messages/{en,fi,et}.json`. **All user-visible PM strings MUST go through `useTranslations`/`getTranslations` — no hardcoded copy.** Pages are dynamically rendered (cookie read). TO SWITCH TO `/[locale]` LATER (when OpenNext supports Next 16 proxy): see the header comment in `routing.ts` — re-add `defineRouting`+`createNavigation`+`src/proxy.ts`, move pages under `app/[locale]/`, point `request.ts` at `requestLocale`; components + catalogs stay unchanged.
- **PM auth is built (reuse `src/lib/auth`, don't reinvent).** Email+password, sessions in KV `SESSIONS`. `lib/auth/password.ts` = PBKDF2-HMAC-SHA256 via Web Crypto (`crypto.subtle`) — bcrypt/argon2 native libs DON'T run on Workers, so don't add them; hash string is self-describing `pbkdf2$<iter>$<saltB64>$<hashB64>`, verify is constant-time. `lib/auth/session.ts` = opaque random session id in an httpOnly+Secure+SameSite=Lax cookie (`bizbee_session`) → KV record `{userId,createdAt,expiresAt}` keyed `session:<id>`, 7-day TTL (KV GCs expiry); `createSession`/`getSession`/`destroySession`. `lib/auth/user.ts` = `userCount`/`hasAnyUser`/`findUserByEmail`/`findUserById`/`getCurrentUser`/`createUser`. `lib/auth/validation.ts` = locale-agnostic error *keys* (min password 10) resolved against the `auth.errors.*` i18n namespace. **First-registrant rule:** self-registration (`/register`) is open ONLY while `users` is empty → that user becomes SuperAdmin with `canInvite`; afterwards the route renders a closed state and new users must come via the invite flow. Login returns a single generic `invalidCredentials` (no account-existence leak). Server actions live next to the pages under `src/app/(auth)/`; logout is `lib/auth/logout-action.ts`. The home page (`src/app/page.tsx`) is auth-gated (redirects signed-out users to /login or /register). TS gotcha: Web Crypto `BufferSource` typing needs `Uint8Array<ArrayBuffer>` annotations (workerd lib types reject plain `Uint8Array`). Verify auth via build (tsc + next build + opennextjs-cloudflare build) — no real D1/KV here.
