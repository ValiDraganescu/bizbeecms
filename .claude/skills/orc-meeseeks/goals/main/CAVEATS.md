# Caveats — main
Read every line before working. Each entry was learned the hard way by a previous Meeseeks.

- The repo was bootstrapped from scratch as a git repo by the loop driver (baseline commit `38a2377`). Not a pre-existing codebase.
- Stack is confirmed Cloudflare-native: Next.js on Cloudflare Workers (OpenNext), D1 for data, email+password auth with sessions in D1/KV, Site deploys via Cloudflare API. Do NOT introduce non-Cloudflare infra.
- `../aicms` is a sibling reference project (its own git repo) — read it for patterns, never edit it. NOTE: aicms uses Postgres (`pg`) + Resend, NOT Cloudflare D1/Workers, so it is NOT a deploy reference. Mine it only for AI-agent / server-side component-rendering tricks, not infra.
- The PM app lives at `ProjectManager/` (its own npm package, NOT a workspace). It has its own `node_modules`, `package.json`, `.gitignore`. Run all PM commands from inside `ProjectManager/`.
- PM stack pinned: Next.js 16.2.9 + `@opennextjs/cloudflare` ^1.19 + wrangler ^4. Build = `next build`; Cloudflare bundle = `npx opennextjs-cloudflare build` (emits `.open-next/worker.js`). `.open-next/` and `.next/` are gitignored.
- **PM i18n is `next-intl` v4, COOKIE-BASED (no proxy/middleware), NOT path-prefix routing.** WHY: Next 16 renamed middleware→`proxy` and made it Node-runtime only; the current OpenNext Cloudflare adapter can't bundle a Node proxy, and Next 16 hard-rejects `runtime: "edge"` on a proxy ("Proxy does not support Edge runtime"). So `/[locale]` path routing builds with `next build` but FAILS `opennextjs-cloudflare build` (the deploy gate). Cookie-based passes both. Refs: workers-sdk#13755, opennextjs-cloudflare#962. Architecture: `src/i18n/routing.ts` (locale source of truth: EN/FI/ET, `localeNames`, `LOCALE_COOKIE=NEXT_LOCALE`), `src/i18n/request.ts` (`resolveLocale()` = cookie → Accept-Language → default), `createNextIntlPlugin` in `next.config.ts`, root layout wraps `NextIntlClientProvider` + `getLocale()`, `LocaleSwitcher` sets the cookie + `router.refresh()`. Catalogs: `messages/{en,fi,et}.json`. **All user-visible PM strings MUST go through `useTranslations`/`getTranslations` — no hardcoded copy.** Pages are dynamically rendered (cookie read). TO SWITCH TO `/[locale]` LATER (when OpenNext supports Next 16 proxy): see the header comment in `routing.ts` — re-add `defineRouting`+`createNavigation`+`src/proxy.ts`, move pages under `app/[locale]/`, point `request.ts` at `requestLocale`; components + catalogs stay unchanged.
- `wrangler.jsonc` needs `nodejs_compat` AND `global_fetch_strictly_public` compat flags for OpenNext. D1/KV bindings are stubbed as commented TODO there — uncomment + fill IDs when the D1 task runs.
- `next build` auto-rewrites `tsconfig.json` (jsx→react-jsx, adds `.next/dev/types`). This is expected; don't fight it.
- Could NOT verify a real `wrangler deploy` (no Cloudflare account/auth in this env). The build pipeline is verified; actual deploy is a later, auth-dependent task.
- **UI design rules (user-mandated, binding for ALL UI work in PM and CMS):**
  - Use **Tailwind CSS** for styling.
  - Implement a **proper theme with light AND dark modes**.
  - Theme colors MUST be **named by purpose, not by color name** (e.g. `surface`, `surface-muted`, `foreground`, `foreground-muted`, `border`, `primary`, `primary-foreground`, `danger` — NOT `blue-500`, `gray-200`). Define them as CSS variables / Tailwind theme tokens with light+dark values; components reference ONLY these purpose tokens, never raw color names or hex.
  - **Composable components for everything.** Prefer component composition over prop-heavy APIs — e.g. `<Table><TableHeader>…</TableHeader><TableBody>…</TableBody></Table>` rather than passing loads of data/config as a single component's props.
- **Localization rules (user-mandated, binding for ALL UI work):**
  - **ProjectManager UI** must be localized in **English, Finnish, and Estonian** (these three are the fixed PM admin-UI locales). Build every page/component i18n-aware from the start — no hardcoded user-visible strings; all copy goes through the i18n layer. Pick one approach and stay consistent (e.g. `next-intl`); record the choice. Locale should be selectable and persisted (and may default from the browser).
  - **CMS** must support the **same three UI locales (English, Finnish, Estonian)** for its own admin/UI chrome, AND **additionally** allow configuring **any number of languages for the user-facing content** of a Site (the published content locales are data-driven per Site, separate from the fixed admin-UI locales).
  - Keep the two concepts distinct: **UI/admin locale** (fixed set EN/FI/ET) vs **content locale** (CMS only, arbitrary, configured per Site).
- PM data layer is **drizzle-orm with the `drizzle-orm/d1` driver** (NOT pg). Schema = `ProjectManager/src/db/schema.ts`; client = `getDb()` in `ProjectManager/src/db/index.ts` (uses `getCloudflareContext({async:true}).env.DB`). Migrations live in `ProjectManager/migrations/` (committed). Workflow: `npm run db:generate` (drizzle-kit) then `wrangler d1 migrations apply bizbeecms [--local|--remote]` (scripts: db:migrate / db:migrate:local). `drizzle.config.ts` uses dialect `sqlite`, driver `d1-http`.
- `wrangler.jsonc` D1 `DB` (`database_id`) and KV `SESSIONS` (`id`) hold **placeholder zero-ids** — no Cloudflare auth in this env to `wrangler d1 create` / `kv namespace create`. Replace with real ids before deploy. `wrangler types` (`npm run cf-typegen`) runs OFFLINE and types `DB: D1Database` / `SESSIONS: KVNamespace` into the (gitignored, regenerable) `cloudflare-env.d.ts` — regenerate it after any wrangler.jsonc binding change.
- Could NOT run real D1 queries or `wrangler d1 migrations apply` (needs Cloudflare auth). Verify D1 work via `npx tsc --noEmit` + `npm run build` + `npx opennextjs-cloudflare build` + checking the generated migration SQL.
- **PM UI foundation is built (use it, don't reinvent).** Tailwind v4 (`@tailwindcss/postcss` in `ProjectManager/postcss.config.mjs`; CSS-first config — NO `tailwind.config.js`). All theme tokens live in `ProjectManager/src/app/globals.css` (`@theme inline`). Purpose tokens available as utilities: `surface`, `surface-muted`, `surface-raised`, `foreground`, `foreground-muted`, `border`, `primary`, `primary-hover`, `primary-foreground`, `danger`, `danger-hover`, `danger-foreground`, `success`, `ring`. Use ONLY these in markup — never `bg-blue-500`/hex. To add/adjust a color, edit the three blocks in globals.css (`:root` light, `[data-theme="dark"]`, and the `@media (prefers-color-scheme: dark) [data-theme="system"]` block) — keep all three in sync.
- Theme: `<html data-theme="light|dark|system">` drives colors. `ThemeProvider`/`ThemeScript` (`src/components/theme/theme-provider.tsx`) + `ThemeToggle`. `ThemeScript` is a blocking inline script in `<head>` (layout.tsx) that prevents FOUC — keep it; `<html>` has `suppressHydrationWarning`. `useTheme()` gives `{theme, resolvedTheme, setTheme}`; theme persists to localStorage key `bizbeecms-theme` and follows OS when set to `system`.
- Composable base components live in `ProjectManager/src/components/ui/` (barrel `@/components/ui`): `<Button>` (variants primary/secondary/ghost/danger), `<Card>` family, `<Table>` family (Table/TableHeader/TableBody/TableRow/TableHead/TableCell), form `<Field>` family (Field/FieldLabel/FieldHint/FieldError/Input/Textarea/Select). Composition-over-props — build pages from these, don't pass big config-prop objects. Build new shared UI the same way and export from the barrel.
- Root `DESIGN.md` is the project design-system doc (impeccable plugin seed). Read it for the visual north star ("Control Room" — calm/precise admin tooling) before UI work; keep new UI consistent with it.
- The CMS app lives at `CMS/` — its OWN npm package (own node_modules/package.json/.gitignore), same pattern as ProjectManager. Run CMS commands from inside `CMS/`. It mirrors PM's OpenNext/wrangler wiring (next 16.2.9, @opennextjs/cloudflare ^1.19, wrangler ^4); dev port is 3602 (PM is 3601). Its `wrangler.jsonc` deliberately has NO D1/KV bindings — the CMS is the plain default Next install this milestone; the PM deploy step will override the Worker `name` per-Site via the Cloudflare API.
