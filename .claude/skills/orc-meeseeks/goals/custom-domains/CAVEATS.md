# Caveats — custom-domains
Read every line before working. Each entry was learned the hard way by a previous Meeseeks.

- Project is Cloudflare-only. Custom hostnames must attach to the `bizbeecms.com` zone via Cloudflare APIs / wrangler — no external DNS/proxy infra.
- The SSO own-zone check already accepts any `*.bizbeecms.com` host, so `<slug>.site.bizbeecms.com` needs NO allowlist code change for SSO. Do not "fix" what isn't broken. Verify SSO end-to-end after each domain change rather than editing the allowlist speculatively.
- Don't loosen the SSO host allowlist. The 4-label + account-suffix anchoring (commits 01e7341, 9b1898d) exists to stop attacker-account lookalikes. Any change here is a security regression risk — verify it still rejects bad hosts.
- `npx opennextjs-cloudflare build` is the deploy gate; NEVER run it while `npm run dev` (port 3601) is running — it corrupts `.next` and 500s the dev server. Stop dev first.
- Per PM memory: REST only, no server actions (they 500 on Workers). Workers PBKDF2 capped at 100k iterations.
- Non-Pages Worker custom domain = a `routes:[{pattern,custom_domain:true}]` entry in its `wrangler.jsonc`; Cloudflare attaches the hostname+cert on `wrangler deploy`. workers.dev subdomain stays reachable unless explicitly disabled — keep it as a fallback.
- `cmsWorkerUrl()` (worker-url.ts) NO LONGER derives the account subdomain from APP_ORIGIN — it builds the workers.dev URL from `WORKERS_DEV_SUFFIX` (hosts.ts). So APP_ORIGIN is now free to be a custom domain. Don't "restore" the old APP_ORIGIN parsing. When sites move to `<slug>.site.bizbeecms.com`, update cmsWorkerUrl to return that host (there's a `ponytail:` comment marking the spot).
- Per-site CMS hostnames `<slug>.site.bizbeecms.com` are served by the ROUTER (router/src/index.ts), which derives the slug from the leftmost subdomain label — NO HOST_MAP KV entry is written for them (contrast: customer-owned custom hostnames DO use HOST_MAP). The deployer was NOT changed; internal worker name stays `bizbeecms-cms-<slug>`. Don't add a per-site custom-hostname attach for the `.site.` scheme — it's redundant.
- The `*.site.bizbeecms.com/*` router route needs a `*.site.bizbeecms.com` WILDCARD cert on the bizbeecms.com zone to terminate TLS live. Confirm that cert/advanced-cert exists before/at first deploy — code-only changes won't fix a missing cert.
- `cmsWorkerUrl()` now returns `https://<slug>.site.bizbeecms.com` (derives slug by stripping `CMS_WORKER_PREFIX` from the worker name). It returns `null` for any name not starting with `bizbeecms-cms-`, which is the guard that prevents `.workers.dev`/other strings leaking into user-facing links. Don't "restore" the workers.dev URL here. `hosts.ts` is the single source of truth: `SITE_HOST_SUFFIX`, `siteUrlForSlug()`, `PM_ORIGIN`.
- Deployer's `PM_ORIGIN` is now a COMMITTED `vars` entry in `deployer/wrangler.jsonc` (= `https://manager.bizbeecms.com`), NOT just a hand-set secret. `deployer/src/index.ts:262` still falls back to `PM_CALLBACK_ORIGIN` if unset. Verify with `cd deployer && npx wrangler deploy --dry-run` — it prints the bound vars. (Dry-run builds the container image; needs Docker available.)
- **CUTOVER STATE (2026-06-19): workers.dev is decommissioned from all user-facing paths but deliberately still reachable.** All user-facing links/vars point at custom hosts (`manager.*`, `deployer.*`, `<slug>.site.*`). The 4 remaining `.workers.dev` code refs are intentional and MUST stay: (1) `WORKERS_DEV_SUFFIX`/`ACCOUNT_WORKERS_SUBDOMAIN` (hosts.ts) feed ONLY the SSO `isOwnCmsWorker` allowlist (`cms-sso.ts:39`) — deleting them loosens/breaks the allowlist; (2) `cloudflare.ts` `enableWorkerSubdomain` keeps the raw-upload path reachable; (3) `router/src/index.ts:43` uses the workers.dev name as the INTERNAL origin fetch for `<slug>.site.*` (transport, not a user link). Do NOT "finish decommissioning" by deleting these — they are the fallback + transport layer. The workers.dev subdomain itself is NOT disabled (cheap fallback). Goal is code-complete; only live deploy + `*.site.bizbeecms.com` wildcard cert remain (infra).
- `DEPLOYER_URL` is read purely from the Worker env var (`bag.DEPLOYER_URL`); there is NO default in `hosts.ts`. Flipping the wrangler var is enough on the PM side. Contrast: `APP_ORIGIN` IS parsed by `worker-url.ts:30` to derive `.workers.dev` CMS URLs and returns null for non-workers.dev hosts — flipping APP_ORIGIN to a custom domain breaks `cmsWorkerUrl()`. Audit callers before flipping APP_ORIGIN.
