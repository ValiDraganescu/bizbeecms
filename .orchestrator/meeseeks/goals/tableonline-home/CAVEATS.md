# Caveats ‚Äî tableonline-home
Read every line before working. Each entry was learned the hard way by a previous Meeseeks.

- Content/theme/page work goes through the HTTP MCP at `http://localhost:3602/mcp` (bearer token in repo-root `.mcp.json`, key `local-site`), NOT direct DB edits. Call `get_authoring_guide` before block-tree edits ‚Äî `update_page_blocks` expects the FULL current tree back, so `get_page` first or you wipe sections.
- The dev server must be running for MCP calls (`npm run dev` in `CMS/`, port 3602). If :3602 is down, start it; don't switch to build mode. NEVER run `npx opennextjs-cloudflare build` while dev is running ‚Äî it corrupts .next.
- Theme tokens are semantic (surface/foreground/primary/‚Ä¶): hex values are accepted; change tokens, don't hardcode colors in components.
- Collections are `content_*` D1 tables created via the fenced `create_collection` tool (100-table cap). Query/filter via `query_collection` and List `listSource.filter`.
- Killing terminals/processes is dangerous (you may kill your own session or the dev server) ‚Äî check what a PID is before killing.
- `update_theme` MCP tool args are `{ light: {...}, dark: {...} }` at the TOP LEVEL ‚Äî do NOT nest under a `theme` key (that fails with "supply 'light' and/or 'dark' as a token‚Üícolor object"). Verify a theme change by curling `/` and grepping the inline compiled `<style>` block for `--color-<token>:` inside `:root{...}` (light) and `[data-theme="dark"]{...}` (dark) ‚Äî the raw hex you set appears there directly, no build step needed.
- **PAGES AND COMPONENTS BOTH HAVE A DRAFT/PUBLISH SPLIT** (not just theme). `update_page_blocks`/`create_page`/`set_block_props`/`create_list`/`bind_list` all write the page's DRAFT version (`page.draft_version_id` ‚Üí `page_version`); the PUBLIC route (`src/app/[[...slug]]/page.tsx`) renders `page.published_version_id` only. Likewise `update_component`/`create_component` write `component.draft_*` columns; `bind_list`'s prop-declared check (and the public renderer) reads the PUBLISHED `component.props_schema`, not the draft ‚Äî so a freshly-`update_component`'d prop name will make `bind_list` fail with `"prop \"x\" is not declared on the template component"` even though `get_component` (MCP) happily shows you the draft schema. **After any content/component change you want live, publish BOTH**: `curl -X POST http://localhost:3602/api/pages/<id>/publish` (page) and `curl -X POST http://localhost:3602/api/components/<Name> -d '{"action":"publish"}'` (component) ‚Äî both are `requireAdmin` REST routes, but local dev auto-auths as SuperAdmin (`.env.local` `CMS_DEV_SUPERADMIN=1`), so a bare unauthenticated curl works on :3602. No MCP tool exposes publish ‚Äî it's REST-only by design (keeps AI from silently going live).
- `update_page_blocks` MCP tool's page-id argument is named **`id`**, not `page` (unlike `create_list`/`bind_list`/`set_block_props` which use `page`) ‚Äî passing `page` gives `"id is required (use list_pages/get_page to find it)"`.
- `create_collection`/`add_collection_field` reject a field literally named `slug` ‚Äî it's a reserved SYSTEM column every collection already has (along with `status`, `archived_at`, `created_at`, `updated_at`). You CAN still set `slug` and `status` explicitly via `add_collection_item`'s/`update_collection_item`'s `values` object (e.g. `{"slug":"helsinki","status":"published"}`) ‚Äî they're just not declarable as a custom field. Fresh `add_collection_item` rows default to `slug:null, status:"draft"` if you don't pass them (existing `content_restaurants` rows are all `status:"draft"` too and still render fine in a List ‚Äî collection item drafts, unlike page/component drafts, don't block List rendering; only pages/components gate on publish).
- A List's `map` (`create_list`/`bind_list`) can only bind a target prop to an EXISTING collection field ‚Äî there's no way to compute/prefix a value (e.g. `"/" + slug`). For a card that needs a URL built from a field (city href, detail-page href, etc.), give the component a plain `string` prop (not `type:"link"`) and bake the fixed prefix into the HTML template itself, e.g. `href="/{{citySlug}}"` with `map:{citySlug:"slug"}`.
- Batching many sequential `generate_image` MCP calls in one shell loop (`curl` in a `for` loop) hits the Bash tool's ~2-minute default timeout after ~4-5 image generations (~20-25s each) and gets killed mid-request ‚Äî the ones that already completed DID save to the gallery (check `list_assets`), but the loop silently stops. Issue `generate_image` one call per tool-turn (or pass an explicit longer `timeout` per curl AND keep the loop short) instead of a long batched loop.
- **Inconsistent `collection` arg naming across collection MCP tools**: `create_collection`'s `name` (and the tool infers/prefixes it) takes the BARE label (e.g. `"offers"` ‚Üí creates table `content_offers`), but `add_collection_field` and `add_collection_item`'s `collection` arg wants the FULL prefixed table name (`"content_offers"`) ‚Äî passing the bare label to either gets `"collection not found"`. Always pass the bare label only to `create_collection`; everywhere else (`add_collection_field`, `add_collection_item`, `update_collection_item`, `query_collection`) pass the full `content_*` name.
- `add_collection_field` field `type` values are NOT the same vocabulary as component `propsSchema` types: use `"asset"` for an image field on a COLLECTION (not `"image"` ‚Äî that's rejected with `unknown field type`), while a component prop for the same kind of thing uses `type:"image"`. Two different type enums for two different systems.
- **Collection item fields (`add_collection_item`/`update_collection_item` `values`) are single-locale plain strings, NOT translatable locale objects** ‚Äî even though the authoring guide's locale-object instructions apply to component PROPS. Passing `{"en":"...","fi":"..."}` for a collection field like `title`/`description` silently stores the JS-stringified literal `"[object Object]"` (no error!) ‚Äî always pass a plain string for collection item field values; verify with `query_collection` if unsure whether a field is single- or multi-locale (city/offer/restaurant collections all use single-locale `title`).
- `create_collection` accepts a `fields` array INLINE at creation (same `{name,type}` shape as `add_collection_field`, e.g. `type:"asset"` for images) ‚Äî `fields` is actually REQUIRED (`create_collection` with no `fields` key errors `"fields must be an array"`), so just pass the whole schema up front in one call instead of creating empty + looping `add_collection_field` per column.
- When generating kebab-case slugs from names with accented chars (Atelj√©, Sk√∂rd, Milj√∂√∂), transliterate FIRST via `unicodedata.normalize('NFKD', s)` + strip combining marks, THEN lowercase/regex-strip ‚Äî a naive `re.sub(r'[^a-z0-9]+','-', s.lower())` on the raw string silently drops the base letter too (produces `atelj-finne`/`sk-rd` instead of `atelje-finne`/`skord`).
- `update_collection_item`'s JSON-RPC response wraps the tool's own `{"ok":true,...}` payload as an escaped string inside `result.content[0].text` ‚Äî a naive substring check on the raw curl stdout (e.g. grepping for literal `"ok":true`) won't match because of the escaping; parse the JSON properly (or re-`query_collection` afterward) to confirm success instead of eyeballing curl output.
- `content_restaurants.location` is free-text city names, not slugs ‚Äî some values (e.g. "Voru"/V√µru) have no matching `content_cities` entry among the 11 seeded FI/EE cities. When mapping `location`‚Üí`city_slug`, fall back to the nearest seeded city rather than leaving it unmapped/blank (mapped Voru‚Üítartu here); flag any such fallback in the JOURNAL so a future city-data pass can add the missing city properly.
- The media gallery already has a few generic "cinematic restaurant interior" images from earlier runs that were generated but never assigned to a collection item ‚Äî check `list_assets` before spending `generate_image` calls; reusing an unused existing asset is free and instant vs. ~20-25s per fresh generation.
- **PLATFORM FEATURE (2026-07-02): dynamic/param-driven pages are DONE.** A page slug prefixed with `:` (e.g. `:city-slug`, `:restaurant-slug`) is a WILDCARD segment ‚Äî matches ANY concrete path segment there; the matched value is exposed as a route param. A List/binding filter `value` can be `{"param":"city-slug"}` (reads the matched wildcard param) or `{"query":"q"}` (reads a `?q=` URL query param) INSTEAD of a literal ‚Äî resolved fresh per request; if the named param/query is absent this request, that filter clause is silently DROPPED (graceful ‚Äî not "filter on undefined"). Works via `bind_component`/`create_list`/`bind_list`'s existing `filter[].value` arg (already `unknown`-typed, no MCP schema change needed beyond the description). Build city/restaurant/offer detail pages with a `:slug`-prefixed page + these filter refs; build /search with a top-level page reading `{"query":"q"}`. See JOURNAL 2026-07-02 16:15 for the full design + a live-verified smoke test.
- **MCP `create_page` writes the LEGACY `page.blocks` column directly ‚Äî NOT a draft version.** So `get_page` (reads the DRAFT version via `getDraft`) and `bind_component`/`bind_list`/`create_list` (which mutate the draft) see an EMPTY block tree right after a bare `create_page` call, even though the page IS live with those blocks (the public route falls back to `page.blocks` when there's no `publishedVersionId`). WORKAROUND: after `create_page`, call `update_page_blocks` once with the SAME tree (it writes the draft version) BEFORE any `bind_*`/`create_list`/`get_page` call on that page. This is a PRE-EXISTING wiring gap in the AI tool pipeline (create_page vs. the versioned draft system), not something this run fixed ‚Äî flag it if a future run hits confusing "no block with id X" errors right after `create_page`.
- **A NEW page is `publishStatus:"draft"` even after `POST /api/pages/<id>/publish` succeeds.** That REST route only bumps `publishedVersionId` (the versioned draft/publish system); the PUBLIC route (`[[...slug]]/page.tsx`'s `resolvePage`) ALSO gates on the page ROW's `publishStatus` COLUMN ‚Äî a separate, older draft/live toggle that only `create_page`'s `publishStatus` arg (or the admin UI's page-settings toggle, a meta-PATCH) sets. A freshly-`create_page`'d page will 404 on its public URL even after a successful `/publish` call until you ALSO re-call `create_page` with the SAME slug + `publishStatus:"published"` (re-passing the current full block tree ‚Äî `get_page` first ‚Äî is harmless: `create_page`'s upsert overwrites the legacy `blocks` column too, but once a `publishedVersionId` exists that legacy column is unused, the versioned tree wins). `list_pages`'s `publishStatus` field is the tell ‚Äî check it after publishing a NEW page.
- **A List block's `filter` clauses AND together ‚Äî there's no OR across fields.** For "field A OR field B contains X" (e.g. search a restaurant's name OR location), use the query-compiler's separate `search` free-text mode instead (`ListSource.search`, added this run) ‚Äî it LIKEs across every text-typed collection field and OR's the results. `filter` stays for exact/comparison AND-conditions.
- **Route-value refs (`{"param":"x"}`/`{"query":"x"}`) only worked in List/binding FILTER values before this run ‚Äî NOT in a component's plain `props`.** Setting a block prop directly to `{"param":...}` used to render the literal stringified object (`[object Object]`), not the resolved value. Fixed this run (`resolveRouteProps` in `route-params.ts`, wired into `render-page.tsx`'s `hydrateBlockBindings`) ‚Äî a STRING prop set via `set_block_props`/`create_page`/`update_page_blocks` can now ALSO be a route-value ref and gets resolved per-request (missing ‚Üí `""`, not a crash). Use this for page text that should echo the current route/query (e.g. a search page's "Results for '{{q}}'" heading).
- **`{"param":"x"}`/`{"query":"x"}` refs ALSO work as a `bind_component`/`bind_list` FILTER value directly against `content_cities.slug`** ‚Äî so instead of resolving a prop via `resolveRouteProps` you can `bind_component` a block to `content_cities` filtered `slug eq {param:"city-slug"}` and map real fields (`title`‚Üí`cityName`, `image`‚Üí`backgroundImage`) onto it ‚Äî gets you the properly-cased display name AND a real photo per param value, for free, no manual echo-the-slug-as-text needed. Used for `CityHero`/`CityStripHeading` on the `:city-slug` page ‚Äî reuse this pattern for any future `:slug`-page hero/heading that needs the CANONICAL (not raw-param) version of a value.
- **Two different collections can store "the same" city two different ways**: `content_restaurants.city_slug` is an actual slug (data-fixed to match `content_cities.slug` exactly), but `content_offers.city` / `content_events.location` are free-text city NAMES ("Helsinki", "Turku") ‚Äî comparing either against a lowercase route param (`"helsinki"`) needs `op:"like"` (SQLite `LIKE` is case-insensitive for ASCII, no `COLLATE` override anywhere in this repo ‚Äî confirmed by grep) instead of `op:"eq"` (case-sensitive `=`, would silently match zero rows). Check a collection's actual field values via `query_collection` before choosing `eq` vs `like` for a param-driven filter ‚Äî don't assume every "city"-shaped field is a slug.
- **`bind_list`/`create_list` have NO arg for a List's empty-state (`listRole:"empty"`) child** ‚Äî `map`/`filter`/`sort`/etc. only configure the template + query. To add (or verify) an empty-state component, you must `get_page` (full current tree, AFTER any `bind_list` calls ‚Äî binding mutates `listSource`/`listMap` but doesn't touch `children`), append `{id, component, listRole:"empty"}` into that List block's `children` array yourself, then `update_page_blocks` the whole tree back. Do this as ONE LAST pass after all your `bind_list` calls, not interleaved ‚Äî an `update_page_blocks` before binding will just get overwritten-in-spirit (not literally, but it's wasted round-trips) since `bind_list` only touches `listSource`/`listMap`, not `children`.
- **A `bind_component`-bound prop that resolves to NO row (query matched nothing) keeps the component's static `propsSchema` DEFAULT, not blank/empty.** E.g. `CityHero`'s `cityName` default is `"Helsinki"` ‚Äî visiting a param value with no matching `content_cities` row (a typo'd city slug) renders "Helsinki" in the h1, not an empty string. This is the existing documented graceful-binding contract (empty result ‚Üí prop stays at its default), not a bug ‚Äî just non-obvious that "stays blank" in the docs actually means "stays at whatever the default already was." If a future page's default placeholder is misleading in this fallback case, consider setting the default to something clearly generic (e.g. "this city") rather than a real city name.
- **NESTED wildcard segments under a STATIC parent page work with ZERO new platform code** (confirmed 2026-07-02: `/offers/:offer-slug` under a static `offers` index page). `resolvePage` in `[[...slug]]/page.tsx` walks the tree level-by-level via `parentPageId`/`matchSlugSegment`, generically ‚Äî it was never special-cased to top-level wildcards, so nesting "just works" at any depth. The ONE hard requirement: `create_page`'s `parentSlug` arg needs the parent page to ALREADY EXIST (`"parent page \"X\" not found"` if the child is created first) ‚Äî create the static parent page (and `update_page_blocks`+publish it) BEFORE the wildcard child. `list_pages` already had a precedent for this (`about`‚Üí`home` nesting) that a previous run apparently didn't notice as proof.
- **`bind_component`'s MCP args are FLAT, not nested** ‚Äî `{page, block, collection, filter, sort, map}` (or `source`+`request`+`params`+`map` for an API source), NOT `{page, blockId, source:{collection,filter,sort}, map}`. Passing a nested `source` object (mirroring how a List's `listSource` looks) gets `"block (id) is required"` because the tool never saw a top-level `block` key. Always `curl tools/list` and read a tool's actual `inputSchema` before guessing its shape from a sibling tool's convention.
- **`content_offers.restaurant` / `content_events.restaurant` are FREE-TEXT NAMES that DON'T exist as rows in `content_restaurants`** (confirmed: none of "Kustavin Kipin√§"/"Wohls G√•rd"/"Ravintola Siuntio"/"LOBO"/"White Lady" match any of the 26 seeded restaurant names) ‚Äî offers/events were seeded from a different informal restaurant roster than the restaurants collection, so there's no relation to bind a real restaurant detail/book link through. Added `content_offers.restaurant_slug` (slugified from the free-text name, same NFKD transliteration recipe as the restaurant data-fix) so the offer page's Book CTA can still point at `/book?restaurant={slug}` consistently with every other card on the site ‚Äî it just won't resolve to a REAL `content_restaurants` row. A future task tightening this (making offers/events reference real restaurants) would need to either reseed offers/events against real restaurant names, or add a `content_restaurants` fallback-insert for the 5 missing names.
- **Publishing a component right after `create_component` (brand new name) correctly reports `{"published":false}`** ‚Äî NOT an error. A new component's artifact is written straight to the LIVE columns (no draft), so there's nothing to publish; `published:false` just means "no-op, already live." Only `update_component` on an EXISTING component creates a draft that a subsequent publish call actually flips.
- **`publicSubmissions` on a collection is OPERATOR-ONLY, no MCP tool** (by deliberate design ‚Äî keeps AI from silently opening a collection to the public). Enable it with a bare REST PATCH: `curl -X PATCH http://localhost:3602/api/collections/<content_table> -d '{"_op":"set_public_submissions","enabled":true}'` ‚Äî local dev auto-auths SuperAdmin so an unauthenticated curl works, same pattern as the page/component publish REST routes. `create_form` errors with this exact fix quoted if you skip it.
- **`create_form`'s `child` component's `<input name=‚Ä¶>` values must match the target collection's field names BY NAME** (snake_case if that's how you declared the collection fields) ‚Äî there's no `map` argument like `bind_list`/`bind_component` have. Name the component's inputs after the collection schema up front (e.g. `party_size`, `customer_name`) rather than guessing camelCase and having to rename later; `create_form`'s response echoes the expected field list so you can diff against your component's inputs immediately.
- **`create_form` auto-generates block ids**: the Form block itself gets `id` = its `name` arg you'd expect, but in practice came back `"Form-1"` (tool-assigned, not caller-chosen), and the nested `child` component instance gets id `"<formBlockId>-child"` (e.g. `"Form-1-child"`) ‚Äî `get_page` after `create_form` to read these ids back before trying to `set_block_props` on the child (don't guess the id).
- **Prop-level route-value refs (`{"query":"x"}`/`{"param":"x"}`, from the `resolveRouteProps` feature) work via `set_block_props` on ANY string prop of ANY component instance**, not just page-level heading props (`SearchHeader`'s `query` was the only prior example) ‚Äî used here on `BookingForm`'s `restaurantSlug`/`restaurantName` props to echo `/book?restaurant=x` into both a visible heading AND hidden form-field defaults simultaneously, confirming the mechanism is fully general per-prop, not tied to any specific component.
- **A visitor form submission's success path is a real HTTP flow**: `POST /api/forms/submit` (with hidden `__bb_page`/`__bb_block` routing fields the Form block auto-injects) returns `303 See Other` ‚Üí `Location: <redirect-path>?bb_form=ok` on success (appends its own `bb_form=ok` query flag to whatever `redirect` you configured, doesn't just redirect bare) ‚Äî test a form's true end-to-end wiring by POSTing to this endpoint directly and then `query_collection`-ing the target table, don't just eyeball that the page renders inputs.
- **`query_collection`'s MCP arg is `filters` (PLURAL)** ‚Äî NOT `filter` (singular), unlike `bind_component`/`bind_list`/`create_list` which all use `filter` (singular). Passing `filter` to `query_collection` is silently accepted (no schema-validation error) and just returns ALL rows unfiltered, which looks exactly like a query-compiler bug if you're mid-debugging something else. Always `curl tools/list` and check a tool's OWN `inputSchema` before assuming filter-arg naming is consistent across the collection/binding tool family ‚Äî it isn't.
- **WILDCARD-UNDER-WILDCARD nested pages (e.g. `:restaurant-slug` under `:city-slug`) work with ZERO new platform code**, confirmed live end-to-end (2026-07-02) ‚Äî `resolvePage`'s per-level walk in `[[...slug]]/page.tsx` is fully generic regardless of how many ancestor levels are themselves wildcards, exactly like the earlier wildcard-under-STATIC precedent (`:offer-slug` under `offers`). BUT: a fresh `create_page` + `bind_component` sequence on ANY new page (nested or not) is a NO-OP live until you ALSO `POST /api/pages/<id>/publish` ‚Äî if you skip it, `page.published_version_id` stays NULL and the public route silently falls back to the LEGACY `page.blocks` column (the pre-binding tree), so EVERY route/param combo renders the same static component defaults. This looks exactly like "the route-param/binding mechanism is broken" (a plausible-sounding platform bug) when it's actually just a missed authoring step already documented elsewhere in this file (see the draft/publish CAVEATS entries) ‚Äî burned real debugging time chasing a phantom platform bug before checking `published_version_id` in D1 directly (`npx wrangler d1 execute bizbeecms-cms --local --command "SELECT published_version_id FROM page WHERE slug=...`). Always publish immediately after the FIRST `bind_component`/`bind_list` call on a brand-new page, before doing ANY live smoke test.
